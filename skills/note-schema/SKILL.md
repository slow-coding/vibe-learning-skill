# Note Schema

Obsidian vault data model: type system, frontmatter conventions, naming rules, directory structure, and body structure patterns. This is the foundation layer — domain-specific schemas (papers, games, etc.) extend this.

**Trigger:** Load this skill before creating or modifying any card.

---

## 1. Card Creation Workflow

**Language rule:** Cards match the user's language. Content, frontmatter values, and filenames follow the language of the query. Field names in frontmatter default to English (type, aliases, domain, etc.) unless the user's vault already uses a different convention — match existing conventions, don't introduce new ones.

1. **Determine type** — identify which card type (see section 2)
2. **Match template** — read the corresponding template file in `templates/`
3. **Fill frontmatter** — follow template fields + generic rules below
4. **Write body** — follow type-specific body patterns (section 5)
5. **Link** — add wikilinks, update referenced cards, check hierarchy
6. **Verify** — confirm no new issues introduced

**Never define frontmatter fields from intuition. All fields must come from templates or this skill.**

## 2. Type System

### Entity Types

| Type | Required Fields | Description |
|------|----------------|-------------|
| person | name | Real people |
| character | name, appears_in | Fictional characters |
| organization | org_type | Companies, teams, communities |
| tech | domain | Languages, frameworks, tools, protocols |
| concept | — | Theories, methodologies, terms |
| product | domain | Software products, services, platforms |
| game | developer, series | Games — see `game-card` skill for full schema |
| hardware | category | Consoles, devices |
| location | — | Cities, locations |
| mechanic | origin_game | Game mechanics (create when 3+ cards reference it) |
| worldbuilding | — | Game world settings |

| Type | Required Fields | Description |
|------|----------------|-------------|
| event | date | Point-in-time events |
| decision | date | Decision rationale |
| project | start_date | Time-bounded work units |

## 3. Frontmatter Rules

### Generic Rules

1. **No wikilinks in frontmatter** — `[[Link]]` goes in body only. Frontmatter values are plain text.
2. **No comma-concatenation** — Don't use `"A, B, C"` for multiple values.
3. **Arrays use inline format** — `platform: ["PS5", "Switch"]`. Not YAML block format (`- item`).
4. **aliases** — Recommended for every card. Format: `aliases: ["A", "B", "C"]`. Include full names, translations, abbreviations.
5. **Remove empty fields** — `aliases: []` or `tags: []` should be deleted entirely.

### Prohibited Fields

| Field | Reason |
|-------|--------|
| `tags` | Often auto-injected by sub-agents; remove on sight |
| `thumbnail` | Auto-generated by Obsidian plugins; manual setting causes conflicts |
| Wikilinks in frontmatter | Use plain text only |

### Type-Specific Fields

For **paper** cards, see `research-paper` skill.
For **game** cards, see `game-card` skill.

Other types:

**tech:**
```yaml
type: tech
domain: "AI | Frontend | Backend | ..."
purpose: "What problem it solves"
source: "Docs/paper/blog URL"
```

**organization:**
```yaml
type: organization
org_type: "Tech company | Game studio | Community | ..."
city: ""
parent_company: ""
```

## 4. File Naming

### General Rules

- No spaces — use hyphens `-`
- Chinese filenames are allowed
- No made-up terminology — use objective descriptions

### Entity Suffixes

Disambiguate entity types with parenthetical suffixes when needed:

| Type | Suffix | Example |
|------|--------|---------|
| Company/Org | (organization) | Anthropic (organization).md |
| Person | (person) | Harrison Chase (person).md |

### Paper Naming

Full English paper title as filename. No parenthetical Chinese. Title variations go in `aliases`.

### Game Naming

Chinese name as filename (e.g. `Final Fantasy 7.md`). Original and alternate names go in `aliases`.

## 5. Body Structure Patterns

### Generic Rules

- No blank lines between table header and separator
- No "known/unknown" headings
- Links appear naturally in prose, not mechanical lists
- Use **mermaid** for diagrams — no ASCII box characters (`┌┐└┘├┤`). Inline arrows (A → B) and tree listings are fine.

### Type-Specific Body Templates

**problem:**
1. Problem — one sentence
2. Background — why now
3. Clues — related card links
4. Current thinking — leaning direction
5. Blind spots — missing info

**decision:**
1. What was chosen
2. What was not chosen
3. Why
4. Expected outcome
5. Actual result (fill in later)

**mechanic:**
1. Mechanism description — how it works, core variables
2. Evolution timeline — key mutation nodes from origin to now
3. Cross-series impact — table of series that adopted/adapted it
4. Agency spectrum position
5. Connections

**worldbuilding:**
1. World overview
2. Key elements
3. How it affects gameplay
4. Appearance timeline
5. Connections

**summary:**
1. Topic definition
2. Categorized links to all related cards
3. Current understanding / open questions

## 6. Directory Convention

The type system and frontmatter rules are **independent of directory structure**. Organize your vault however you like — by topic, by type, flat, or nested. The only hard rule is that every `.md` file with frontmatter must start with `---` as the very first character.

Below is an example layout (the one we use), not a requirement:

```
vault/
  cards/            # All index cards
    coding/         # Tech, frameworks, papers, projects
    game/           # Games, series, characters, hardware
    life/           # Personal notes, finance, planning
    misc/           # Everything else
  bases/            # Obsidian Bases (.base files)
  templates/        # Card templates
```

If your vault is organized differently, just pass your directories to the validation script. The `--schema` flag lets you define your own types and field rules without modifying the script.

## 7. Pitfalls

- **Sub-agents don't read templates** — When delegating card creation, include complete frontmatter field definitions in the context. Don't expect sub-agents to read templates themselves.
- **Sub-agents auto-add tags** — OpenAI-series agents frequently inject `tags:` fields. Check and remove after creation.
- **Sub-agent path drift** — Always provide full absolute paths to sub-agents, never just `cards/`.
- **iCloud + Obsidian write conflicts** — External writes can be overwritten while Obsidian is running. Use Obsidian API or close Obsidian before writing.
- **Feature field duplicate injection** — Obsidian controls may inject duplicate `feature` fields. Check after writing.
- **Chinese filename search unreliability** — `search_files` may not find Chinese filenames reliably. Use `find` to verify file existence.

## Scripts

### validate_frontmatter.py

Frontmatter validation and auto-fix tool. Scans any directory for `.md` files.

```bash
# Validate all .md files in a directory
python scripts/validate_frontmatter.py <your-vault-dir>/

# Validate only paper cards
python scripts/validate_frontmatter.py <your-vault-dir>/ --type paper

# Validate game cards
python scripts/validate_frontmatter.py <your-vault-dir>/ --type game

# Use your own type definitions and field rules
python scripts/validate_frontmatter.py <your-vault-dir>/ --schema my-schema.json

# Auto-fix H1-before-frontmatter issues
python scripts/validate_frontmatter.py <your-vault-dir>/ --fix

# JSON output for programmatic use
python scripts/validate_frontmatter.py <your-vault-dir>/ --json

# Verbose: show clean files too
python scripts/validate_frontmatter.py <your-vault-dir>/ -v
```

**Built-in defaults** cover `paper` and `game` types. If your vault uses different types or field names, create a schema file:

```json
{
  "required": {
    "paper": ["type", "paper_category", "paper_authors"],
    "note": ["type", "created"]
  },
  "deprecated": {
    "paper": ["paper_year", "citations"],
    "general": ["tags"]
  },
  "arrays": {
    "paper": ["paper_authors", "aliases"],
    "note": ["tags"]
  }
}
```

**Checks performed:**
1. File starts with `---` (frontmatter delimiter)
2. Exactly 2 `---` delimiters (opening + closing)
3. No H1 before frontmatter (silently breaks Obsidian Bases)
4. Required fields present (per type, from built-in or --schema)
5. Deprecated fields detected (from built-in or --schema)
6. No wikilinks in frontmatter values

**Auto-fix:** `--fix` repairs H1-before-frontmatter by moving H1 after the closing `---`. Verifies delimiter count after fix.

**Exit code:** 0 if all files pass, 1 if any issues found.
